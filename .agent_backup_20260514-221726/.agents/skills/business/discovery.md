# Product Discovery Skill Pack

This skill pack contains frameworks for discovering opportunities, ideating, and validating assumptions. Use these tools during the early stages of product development or when exploring new features.

## 1. Opportunity Solution Tree (OST)
Developed by Teresa Torres. Use this to visually map out the path from a desired outcome to user opportunities, solutions, and experiments.
- **Outcome**: The business value you want to drive (e.g., Increase activation rate).
- **Opportunity**: User needs, pain points, or desires that can drive the outcome.
- **Solution**: Ideas for how to address the opportunity.
- **Experiment**: Tests to validate the solution before building.
*Prompt Pattern:* "Help me build an Opportunity Solution Tree for [Business Outcome]."

## 2. Assumption Mapping
Identify and prioritize risky assumptions across four key dimensions before building.
- **Value Risk**: Will customers choose it or buy it?
- **Usability Risk**: Can users figure out how to use it?
- **Feasibility Risk**: Can we build it with the time/skills/technology we have?
- **Viability Risk**: Does this solution work for our business?
*Prompt Pattern:* "What are the riskiest assumptions for [Feature/Product Idea]?"

## 3. Ideation & Brainstorming
Structured brainstorming for ideas or experiments.
- **For Existing Products**: Focus on improving current metrics, reducing friction, or extending value.
- **For New Products**: Focus on finding product-market fit, testing core value propositions, and lean pretotyping.
*Prompt Pattern:* "Brainstorm [ideas/experiments] for [existing product problem / new product concept]."

## 4. Customer Interview Scripting
Create structured interview guides focusing on Jobs-to-be-Done (JTBD).
- Focus on past behavior over future predictions.
- Listen for specific pain points and compensating behaviors.
*Prompt Pattern:* "Create an interview script to understand [target user]'s workflow when doing [task]."

## 🛠️ Implementation Checklist
- [ ] Is the business outcome clearly defined before mapping opportunities?
- [ ] Have assumptions been categorized by Value, Usability, Feasibility, and Viability?
- [ ] Are the riskiest assumptions prioritized for testing first?
- [ ] Are we testing assumptions with lightweight experiments before writing code?
